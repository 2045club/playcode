# AGENTS.md

## 项目概览

本项目默认技术栈如下：

- Next.js
- shadcn/ui
- SQLite

若仓库后续未明确说明，默认采用以下实现约定：

- 使用 Next.js App Router
- 使用 TypeScript
- 使用 Tailwind CSS 作为样式基础
- 使用 shadcn/ui 构建基础组件
- 使用 SQLite 作为本地数据存储

本文件用于约束在本仓库中工作的 AI Agent、自动化工具和协作开发者，确保代码风格、目录结构和实现方式保持一致。

## 工作原则

- 优先保持实现简单、直接、可维护。
- 优先复用已有组件、工具函数和样式变量，避免重复造轮子。
- 优先服务端实现数据读取与写入，减少不必要的客户端状态同步。
- 优先类型安全，避免 `any`、隐式类型和不必要的类型断言。
- 在未被明确要求的情况下，不引入额外大型依赖。

## 目录约定

若项目初始化时未另行约定，推荐采用如下结构：

```txt
app/                  Next.js App Router 页面、布局、路由处理
components/           业务组件
components/ui/        shadcn/ui 基础组件
lib/                  通用工具、常量、服务封装
lib/db/               SQLite 连接、查询封装、schema 相关逻辑
actions/              Server Actions
styles/               全局样式或主题扩展
public/               静态资源
```

补充约定：

- 页面级逻辑优先放在 `app/`
- 通用 UI 组件优先放在 `components/`
- 数据访问逻辑不要散落在页面组件中，应集中在 `lib/db/` 或服务层
- 不要在 `components/ui/` 中放业务逻辑

## Next.js 约定

- 默认优先使用 App Router。
- 默认优先使用 Server Components。
- 仅在以下情况使用 `"use client"`：
  - 需要浏览器事件交互
  - 需要本地状态
  - 需要使用仅客户端可用的 hooks
- 数据获取优先在服务端完成，不要把本可服务端完成的查询下放到客户端。
- 写操作优先通过 Server Actions 或 Route Handlers 完成。
- 若涉及鉴权、数据库、文件系统、密钥等能力，必须放在服务端。

## shadcn/ui 约定

- 优先基于 shadcn/ui 现有组件组合实现页面，不随意重新封装重复组件。
- 新增基础 UI 组件时，优先遵循 shadcn/ui 的命名与目录方式。
- 业务组件可以组合 `components/ui/` 中的基础组件，但不要直接修改生成组件的核心行为，除非确有必要。
- 样式优先使用 Tailwind CSS 工具类；仅在复用价值明确时抽取样式。
- 组件变体优先通过 `class-variance-authority` 或项目现有模式管理。
- 统一使用 `cn()` 一类工具合并 className；若仓库已有同类工具，保持一致。

## SQLite 约定

- SQLite 访问必须仅发生在服务端。
- 不允许在客户端组件中直接访问数据库。
- 所有 SQL 操作必须使用参数化查询，避免拼接 SQL。
- 涉及表结构变更时，必须同步补充迁移方案或初始化脚本。
- 数据层应尽量封装成清晰的方法，不要把原始查询散落在多个页面或组件中。
- 对列表筛选、唯一约束、外键关系等场景，应显式考虑索引和数据完整性。

若项目尚未选定 ORM / Query Builder：

- 默认保持 SQLite 原生访问或轻量封装
- 不主动引入 Prisma、Drizzle 等额外层，除非需求已明确

## 编码规范

- 使用 TypeScript 严格模式思维编写代码。
- 函数组件命名使用 PascalCase。
- 工具函数文件命名可使用 `kebab-case`。
- 避免超大文件；当文件明显承担多个职责时应拆分。
- 避免过度抽象，抽象必须服务于复用或可读性。
- 注释应解释“为什么”，而不是重复“做了什么”。
- 不保留无意义的调试输出、注释代码和未使用变量。

## 数据与状态管理

- 能在服务端完成的数据处理，不放到客户端。
- 能通过 props 传递的状态，不额外引入全局状态。
- 若确需客户端状态，优先使用 React 原生能力。
- 未明确要求时，不引入 Zustand、Redux 等状态管理库。
- 表单提交优先使用 Next.js Server Actions 或与当前项目一致的方案。

## 样式与界面

- 保持简洁、克制、一致的界面风格。
- 优先使用设计 token、CSS 变量和 shadcn/ui 现有机制。
- 保持桌面端与移动端可用。
- 间距、字号、圆角、阴影应尽量统一，不随意发散。
- 不为一次性页面引入复杂动画。

## 依赖管理

- 新增依赖前先确认是否能通过现有能力解决。
- 优先选择社区成熟、维护稳定、体积合理的依赖。
- 若新增依赖，需要保证其与 Next.js、shadcn/ui、SQLite 场景兼容。
- 非必要不要同时引入功能重叠的多个库。

## 测试与验证

完成修改后，至少进行以下验证：

- 类型检查通过
- 关键页面可正常渲染
- 关键交互可正常执行
- 涉及数据库变更时，基本增删改查流程可验证

若仓库已配置相关命令，优先执行：

```bash
npm run lint
npm run typecheck
npm run test
```

若命令不存在，则执行当前仓库已有的等效命令。

## Git 提交约定

- 当一次用户任务已完成、改动范围清晰、且至少完成最小必要验证后，Agent 应默认在结束前完成以下 Git 流程：检查工作区状态、仅暂存本次任务相关文件、创建本地 commit，并在满足推送条件时继续执行 push。这里的“一次任务”指一次完整需求处理，而不是单条命令或中间步骤。若本次任务产生了文件改动，且这些改动可以安全隔离，则未完成本地 commit 视为任务未完成。
- 若本次执行未产生文件改动，或仅进行了分析、排查、答疑，则不强制提交，但应明确说明“未修改代码”或“未提交代码”。
- 提交前必须先检查工作区状态，仅暂存与本次任务直接相关的文件，不得将用户已有的未说明改动、临时文件、日志、构建产物或其他无关改动一并提交。
- 若工作区已有无关改动且无法安全拆分，Agent 不应强行提交；应停止在提交前，并向用户说明原因、受影响范围以及建议的处理方式。
- 提交前应优先执行最小必要验证。默认至少尝试运行 `npm run lint` 与 `npm run typecheck`；若仓库不存在对应命令，则执行等效验证。若验证失败或无法执行，Agent 不得声称“已验证通过”，并应在最终说明中明确指出。
- 提交信息应简洁、可追溯，建议使用 Conventional Commits，例如：`feat(auth): add session bootstrap route`、`fix(workspace): guard missing project id`、`docs(agents): define commit rules`。
- 当一次任务满足提交条件且本地 commit 成功后，若远程仓库已配置、当前分支存在明确 upstream、认证可用、且本次 push 不会连带推送无关的本地提交，则 Agent 应默认立即执行 `git push`，而不是等待用户再次提醒。若用户未指定目标，默认推送当前分支到其对应 upstream。
- 若当前分支是 `main`、`master` 或其他受保护分支、共享主分支，或 push 会带出并非本次任务产生的额外 commit，Agent 不应强行推送；应停止在 push 前，并向用户明确说明原因与风险。
- 若 push 失败，例如鉴权失败、远程拒绝、分支不存在或需要先拉取同步，Agent 应保留本地 commit，清晰报告失败原因，并避免自行执行 `git pull --rebase`、`git push --force`、`git reset` 等高风险操作，除非用户明确要求。
- 除非用户明确要求，Agent 不主动执行 `git merge`、`git rebase`、`git reset`、`git push --force`、`git commit --amend` 等高风险 Git 操作。
- 若任务可拆分为多个可独立回滚的子改动，可分成多个 commit；否则保持一次任务一个 commit，避免制造碎片化提交。
- 若一次任务结束时满足提交与推送条件，Agent 应默认主动完成 `commit` 与 `push`，不应在仅因用户未再次提醒的情况下省略该步骤。
- 若用户最新明确指令与本节约定冲突，以用户指令为准。

## Agent 执行要求

在本仓库中工作的 Agent 应遵循以下规则：

- 修改前先理解相关文件上下文，不盲改。
- 优先局部修改，避免无关重构。
- 不随意改动目录结构、命名体系或基础组件实现。
- 不覆盖用户已有未说明的改动。
- 若发现实现与本文件冲突，应优先遵循仓库中的真实代码与用户最新指令。
- 若需求不明确，优先采用最小可行实现。
